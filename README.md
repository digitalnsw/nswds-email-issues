# NSW Email Toolkit Issue Tracker

This repository keeps track of issues in the [NSW Email Toolkit](https://email.digital.nsw.gov.au).

> **Note:** This repo is for **issues only** – it does not contain the email templates, components, or source code. Those live in separate implementation repositories.

---

## Purpose

This issue tracker is the central place to:

- Report bugs and defects
- Request new features or enhancements
- Ask questions about usage and compatibility
- Track known issues and planned improvements

Using a single tracker helps us keep discussions in one place and makes it easier to see the overall health of the NSW Email Design System.

---

## What to Use This Repo For

Please use this repo’s **Issues** tab for:

- 🐛 **Bugs**
  - Rendering problems in specific clients
  - Accessibility issues
  - Broken layout or components
- 💡 **Feature Requests**
  - New components or patterns
  - Enhancements to existing components
  - Additional documentation or examples
- ❓ **Questions / Clarifications**
  - “How do I…?” questions
  - Best-practice usage
  - Integration questions

If you’re unsure whether something belongs here, open an issue anyway and we’ll help route it.

---

## Before You Open an Issue

1. **Search existing issues**
   - Check both open and closed issues for similar reports or discussions.
2. **Check the design system site**
   - Review the [NSW Email Toolkit documentation](https://email.digital.nsw.gov.au) for existing guidance, patterns, and usage notes.

If you find a related issue, feel free to add a comment instead of opening a new one.

---

## How to Raise an Issue

When opening a new issue, please include:

### For Bugs

- **Summary** – short, clear description of the problem  
- **Steps to Reproduce** – list each step as clearly as possible  
- **Expected Behaviour** – what you thought would happen  
- **Actual Behaviour** – what actually happened  
- **Environment**
  - Email client(s) (e.g. Outlook 365 desktop, Apple Mail, Gmail, etc.)
  - Platform (Windows, macOS, iOS, Android, web)
- **Code Sample / Template**
  - Minimal HTML snippet if possible, or a link to the template
- **Screenshots**
  - Screenshots or screen recordings are extremely helpful

### For Feature Requests

- **Background / Problem** – what you’re trying to achieve  
- **Proposed Solution** – what you’d like to see added or changed  
- **Use Cases** – where and how this would be used  
- **Constraints / Considerations** – performance, accessibility, policy, etc. if relevant

### For Questions

- **Context** – what you’re working on (high level)  
- **What You’ve Tried** – links, docs, or approaches you’ve already looked at  
- **The Specific Question** – as clear and concrete as possible

## Labels & Triage

We use labels to help triage and prioritise issues. Common labels include:

- `bug` – something is broken
- `enhancement` – improvement to an existing feature
- `feature-request` – new capability or component
- `documentation` – docs-related issue
- `question` – general question or clarification
- `accessibility` – accessibility-related issue
- `email-client:<name>` – issues specific to an email client

New issues are periodically reviewed and labelled by maintainers.

---

## Security & Sensitive Issues

Please **do not** report security-sensitive issues (e.g. vulnerabilities, data exposure) in public issues.

Instead, contact the maintainers through the appropriate internal security or support channels for your organisation. If you’re unsure which channel to use, speak with your team’s technical lead or product owner.
